# 회고록 — AI와 함께 웹서비스를 만들며

## 협업 방식
기획부터 디자인, 구현, 배포까지 전 과정을 Claude Code와 대화형으로 진행했다. 코드를 직접 몰라도 "이런 느낌이었으면 좋겠다"는 스크린샷 레퍼런스나 자연어 요구사항만으로 구체적인 화면과 기능이 만들어지는 경험이었다. 완벽한 기획서를 먼저 쓰고 시작한 게 아니라, 대화를 반복하며 요구사항이 점점 또렷해지는 방식에 가까웠다.

## 실제로 구현한 것
- **프론트엔드**: 홈, 작품 목록(장르 필터), 작품 상세, IP 개발(로그인 게이팅 모달), 로그인/회원가입, 관리자 대시보드 등 상호작용 가능한 화면 여러 개를 구현했다.
- **백엔드/API**: Astro의 Content Layer를 이용해 Supabase를 호출하는 데이터 로더를 직접 구성하고, 클라이언트에서도 Supabase JS 클라이언트로 인증·CRUD 요청을 처리했다.
- **DB 연동**: Supabase(PostgreSQL)에 작품·IP·회원 테이블을 설계하고, Row Level Security로 "공개 열람 vs 로그인 전용"을 테이블 단위로 분리해 접근 제어를 구현했다.
- **배포 및 새로운 시도**: 실제 접속 가능한 도메인(Cloudflare Pages)에 배포했고, GitHub Actions로 push 시 자동 빌드·배포되는 파이프라인을 직접 구성했다. 또한 반복되는 "작품 추가" 워크플로우를 Claude Code의 Skill로 만들어, 다음에는 같은 설명을 반복하지 않아도 되게 표준화했다.

## 배운 점
- 완벽한 설계 없이도, 짧은 반복 피드백 루프만으로 서비스가 점점 완성도를 갖춰갈 수 있다는 걸 체감했다.
- 무료 배포 인프라(Netlify, Cloudflare Pages, GitHub Pages)마다 트레이드오프가 다르다는 걸 실제로 한계에 부딪히며 배웠다 — 처음 선택이 항상 최선은 아니고, 필요하면 갈아탈 수 있어야 한다는 것도.
- 반복 작업을 Skill로 문서화해두면 AI와의 협업 자체가 재사용 가능한 자산이 된다는 걸 알게 됐다.

## 어려웠던 점
- 외부 서비스의 계정 연동(예: GitHub 앱 설치, OAuth 로그인)은 API만으로 끝까지 자동화되지 않고, 결국 대시보드에서 직접 클릭해야 하는 지점이 있었다.
- 작업 도중 프로젝트의 Git 저장소 구조가 꼬여 커밋이 엉뚱한 원격 저장소를 향했던 사고가 있었다 — 다행히 백업 덕분에 복구했지만, AI가 하는 작업이라도 중요한 변경 전에는 항상 되돌릴 수 있는 지점을 만들어둬야 한다는 걸 절감했다.
- Supabase의 공개용 키만으로는 테이블 구조 변경 같은 관리자급 작업이 불가능해서, 마지막 SQL 실행은 항상 내가 직접 대시보드에서 처리해야 했다 — AI가 모든 걸 대신할 수 있는 건 아니라는 한계를 명확히 알게 된 지점이었다.

## 종합 소감
AI와의 협업은 "혼자 다 맡기는 것"이 아니라, AI가 반복적인 구현과 실행을 담당하고 사람은 판단과 최종 승인을 하는 구조에 가까웠다. 특히 배포나 계정 권한처럼 되돌리기 어려운 작업은 매번 사람의 확인을 거치는 게 오히려 더 빠르고 안전했다.
